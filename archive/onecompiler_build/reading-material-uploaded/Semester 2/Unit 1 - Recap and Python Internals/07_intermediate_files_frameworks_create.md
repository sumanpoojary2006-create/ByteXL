## Introduction

Asel's first real task at the internship is to add a route to a Flask application. She clones the repository, installs dependencies, and runs `flask run`. Immediately, a `__pycache__` folder fills with dozens of `.pyc` files, and she spots another folder she does not recognize: something generated by a template engine. She wonders whether she should commit these, delete them, or ignore them.

This lesson answers those questions by explaining which intermediate files Python web frameworks generate, why they exist, and what the correct policy is for each. The internals are an extension of what you already know about bytecode and the import system.

## What Happens When a Framework Starts Up

When you run `flask run` or `uvicorn main:app` (for FastAPI), the framework does not just execute your `main.py`. It performs a **startup sequence** that involves importing a large number of Python files, yours and the framework's own.

Because every import compiles and caches bytecode, the first `flask run` on a clean installation generates `__pycache__` folders:
- In your project's source directories
- Inside the Flask or FastAPI package in `site-packages`
- Inside every dependency that gets imported at startup

None of these are your responsibility to manage. They appear automatically, and they speed up every subsequent startup by avoiding recompilation. All of them belong in `.gitignore`.

```
# Standard Python .gitignore entries
__pycache__/
*.pyc
*.pyo
```

## Jinja2 Template Compilation

Flask uses the Jinja2 template engine for rendering HTML. Jinja2 compiles templates (`.html` files) to Python bytecode the first time they are rendered, and caches the result in memory during the process lifetime. In some configurations it can also write compiled templates to a `__jinja_cache__` folder on disk.

This is the same compilation concept as Python source files: Jinja2 parses template syntax, produces a code object, and runs that code object when rendering. The intermediate `.html` files in `templates/` are your source; the compiled form is an implementation detail of Jinja2 that you do not interact with directly.

## FastAPI's Generated Code: Swagger and OpenAPI

FastAPI generates its documentation (the interactive Swagger UI and the `/openapi.json` endpoint) automatically from your Python type hints at startup time. This is not file generation in the traditional sense; the output is kept in memory and served as HTTP responses. No intermediate files appear in your project directory.

What FastAPI does create on disk (if you use it) is a `__pycache__` for your own route files. The generated documentation lives only in memory. This is one of the reasons FastAPI encourages `uvicorn` (an async server) to handle process restarts efficiently: the startup work is repeated each time a worker process starts.

## Migration Files Are Different

Frameworks that include database migration tools, such as Flask-Migrate (built on Alembic) or Django's own migration system, generate Python files in a `migrations/` directory. These are not cached bytecode. They are source files that describe database schema changes, written in ordinary Python, and they absolutely should be committed to version control.

```
migrations/
    env.py
    versions/
        001_add_books_table.py    # COMMIT THIS
        002_add_isbn_index.py     # COMMIT THIS
__pycache__/
    app.cpython-311.pyc           # DO NOT COMMIT THIS
```

The distinction is: if a file is generated to cache something that will be regenerated identically next run, exclude it from git. If a file records an intentional change to your system, commit it.

## Framework Intermediate Files at a Glance

| File type | Source | Commit? |
|---|---|---|
| `__pycache__/*.pyc` | Python's bytecode cache | No |
| `*.pyc`, `*.pyo` | Old-style bytecode files | No |
| `__jinja_cache__/` | Jinja2 compiled templates | No |
| FastAPI OpenAPI JSON | Generated in memory, not on disk | n/a |
| `migrations/versions/*.py` | Migration source files | Yes |
| `.env` files | Local secrets | No |

## Conclusion

Web frameworks like Flask and FastAPI generate `__pycache__` folders for every module they import at startup, Jinja2 may cache compiled templates, and FastAPI generates API documentation in memory. All bytecode caches belong in `.gitignore`. Migration files and other hand-authored generated files are the exception: they represent intentional decisions and should be committed. The final lesson of this unit zooms out from files and processes to the contract that makes all of Python's objects work consistently together: the Python data model and its dunder methods.
